01 — Current state

One runtime model, fed by three authoring models

The "three parallel systems" framing is right — but the engineering reality is sharper.

Items and folders are the same record type, arranged in a tree by a parent pointer (a folder is just a record that holds children). Access is not computed from share records at read time. Instead, a denormalized permission table mirrors one row per item, per user — and that mirror is what every read actually consults. The three "models" below are really three different ways of authoring rows into that one mirror.

model 01

Folder shares

Explicit folder → person invites by email, with an accept / decline lifecycle, plus one shareable invite-link token per folder.

authoring
model 02

Item permission mirror

The real source of truth at read time: a row per (workspace, item, user) with a role, plus a special "everyone" sentinel row for workspace-wide visibility.

runtime truth
model 03

Team folders

A folder flagged non-private becomes visible to the whole workspace — a third path that writes the "everyone" sentinel into the same mirror.

authoring

The load-bearing problem. Because the mirror stores one row per descendant, every membership or visibility change fans out a tree walk that rewrites a row for every item underneath. Add one member to a folder with 10,000 descendants → 10,000 writes. Flip a folder's visibility → delete-and-reinsert the entire subtree. Creating an item walks up its ancestors to copy down inherited rows. A dedicated backfill job exists solely to keep this mirror consistent — which is itself the tell that the model is fragile.

⚠
The read check also walks ancestors when the direct row misses, and the client re-derives the same rules a second time for UI gating — two implementations of one ruleset that can quietly drift.

02 — The thesis

Make access a membership join, not a mirror

The redesign isn't "add a Space table." It's changing what a permission check reads.

Today — per-item mirror

folderflip visibility →
child · child · child …rewrite a row for every descendant
grand-children …and theirs, recursively
membership change = O(subtree) writes
read = walk ancestors until a row hits
→

Proposed — membership join

itembelongs to exactly one Space
Space ⇢ membersone membership list per Space
check = is caller a member?single indexed lookup
membership change = 1 write
read = one join, no ancestor walk

The move: introduce a Space as the single permission boundary. An item belongs to exactly one Space; access is item → Space → "is the caller a member?"

Because the design already mandates single-membership ("a file lives in exactly one Space; assigning it is a move"), this is clean. Membership edits — the common case — touch zero items. The only operation that still rewrites a subtree is a move, and that's a single indexed-column update per item, not a permission-row explosion. Roles collapse from three to two: Owner and Member.

03 — Proposed data model

One container, one membership list, one check

The workspace/org stays as the billing + seats wrapper; Spaces live inside it.

Space

The top-level content container. Carries name, icon, a "default / personal" flag (one per user, archive-not-delete), and a "discoverable" flag for open-to-workspace joining.

Space membership

One row per (Space, user) with role OWNER or MEMBER. This single list replaces the entire per-item mirror.

Space invite + invite link

Pending email invites with accept / decline, and a token link for outside recipients — both re-pointed from folder-level to Space-level.

Item → Space pointer

Each item record gains a spaceId. Denormalized so reads are a single lookup; a move rewrites the pointer down the subtree.

The new access check

The ancestor walk, the "everyone" sentinel, and the per-item mirror all disappear behind one join:

// replaces the recursive ancestor walk + mirror lookup
function access(item, user) {
  const role = membership(item.spaceId, user.id)   // one indexed join
  if (role === 'OWNER')  return read + write + manage
  if (role === 'MEMBER') return read + write
  if (space.discoverable) return joinable    // open-to-workspace
  return forbidden
}
★
Compliance-gated workspaces, sharing seat limits, and inviter-authority re-validation are load-bearing and preserved — re-implemented at the Space level rather than dropped.

04 — Migration

Each existing model maps deterministically

Runs as a versioned, idempotent backfill with a dry-run that reports any access-widening.

Private / owner-only itemsThe owner's default SpaceA one-member Space is simply "private."
Team folders (workspace-visible)A workspace-wide SpaceDiscoverable; all workspace members join.
Shared foldersA new Space per top-level shared folderAccepted recipients become members.
Three rolesTwo rolesOwner→Owner, full-access→Member, viewer→Member. Viewers gain write — flag & confirm.
Pending invites & link tokensSpace invites & Space link tokensExisting tokens must keep resolving.
Legacy public chat-share linkLeft read-only through deprecationSuperseded by query-param share URLs; no data move.
⚠
Sharpest edge — nested shares. A sub-folder can today carry a share distinct from its parent. Single-Space membership can't represent "parent shared with A, child also with B." The likely rule: child shares collapse into the nearest ancestor Space, and any extra members are added there — a conscious widening that the dry-run surfaces for review.

06 — Breakdown

Buildable issues

Sequenced so the new path lands dark behind a dual-read before anything is cut over.

  1. Schema + Space entity. Space, membership, invite, invite-link tables; add the item → Space pointer. No behavior change.
  2. Space-resolving access check. New resolver behind the existing check signature, dual-reading the old mirror as fallback. Keeps all call sites stable.
  3. Default Space at signup. Create a default Space alongside the workspace that's already created per user.
  4. Space CRUD + membership API. Create / rename / archive / invite / remove / role-change, porting compliance, seat-limit, and role-rank guards.
  5. Invite link + redemption. Space link tokens and the workspace-join-then-Space-join path.
  6. Backfill migration. Versioned, idempotent, dry-run with access-widening report; maps the three models and backfills the item pointer.
  7. Move semantics. Assigning an item to a Space = subtree pointer rewrite + a "becomes visible to N members" confirmation.
  8. Sidebar + Library reshape. Space-grouped sidebar; flat Library with a Space filter/column; retire the bucketing logic.
  9. Client permission unification. Replace the duplicated client-side rules with one membership-driven capability source.
  10. Retire the mirror. Once dual-read is proven, drop the per-item mirror, its backfill job, and the old share UI.
  11. Legacy share sunset. Keep the old public chat link read-only, add usage telemetry, set a removal date.

07 — Open questions

To resolve before building

The decisions that change the shape of the work.

  • Nested shared folders — collapse into the ancestor Space, or allow sub-Spaces? Single-membership forbids overlapping shares, so we need the rule.
  • Dropping the viewer role — migrate viewers to Member (they gain write), or keep a transitional read-only flag for workspaces that rely on it?
  • Team folders — one workspace-wide discoverable Space per org, or drop org-wide visibility and make everything an explicit Space?
  • Denormalize the pointer vs. resolve-via-root — accept the per-item pointer (fast reads, subtree rewrite on move)? Recommended: yes, denormalize.
  • Dual-read window — how long do we keep the old mirror alive as a fallback before deleting it?
  • Chats — confirmed exempt from Spaces in the current design; verify they carry no Space pointer at all.
