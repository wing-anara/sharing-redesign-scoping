Authorization model · Drizzle / MySQL

Sharing as a
relation-tuple graph

Replacing per-object ACL denormalization with relationship-based access control: typed relation tuples in one MySQL table, access decided by graph traversal at read time.

Thesis: stop materializing a permission row per (object, subject). Persist edges as tuples — object#relation@subject — and resolve Check(object, relation, subject) as a userset rewrite over the graph, cached.

01 / Write amplification

Denormalized ACLs fan out on every write

Today a grant materializes a permission row for every descendant object. Cost is O(subtree) per mutation.

● Denormalized ACL — row per (object, subject)

subtree of objects under the folder
folder:pharma → 24 descendant objects
permission rows written: 0

● Relation tuple — row per grant

one edge inserted into the tuple store
space:pharma — membership relation
tuples written: 0
run twice — add user:anna, then user:bob

Why it hurts: write amplification is O(subtree) per grant; re-parenting or a visibility flip triggers an unbounded subtree recompute (delete + reinsert every descendant row), and a background reconciliation job has to exist to repair drift. The tuple model writes one row per grant and derives the rest at read time. (The only writes are grants and revokes — one row each. Permission checks are reads; "READ/WRITE" in §03 names the operation being authorized, not a DB write.)

02 / Data model

One table. Every edge is a relation tuple.

Membership, containment, and direct grants share one row shape: object#relation@subject, where the subject is a user or a userset.

member parent viewer (direct)
Permissions aren't stored — they're paths through this graph, computed on demand. The tuples on the right are exactly these edges. (doc:contract is objectType:objectId — readable labels here; stored as two indexed columns, queried by exact match.)

03 / Evaluation

Reads and writes check different relations

Pick a subject. The read check resolves viewer, the write check resolves editor — independently. A viewer-only grant passes one and fails the other.

// relation hierarchy — owner ⊂ editor ⊂ viewer
owner(doc)  ≜ _this
editor(doc) ≜ _this ∪ owner ∪ (parent ↦ member)
viewer(doc) ≜ _this ∪ editor ∪ (parent ↦ viewer)
These rows are reads, not writes. One Check() is a few indexed SELECTs — bounded depth (§05), short-circuited (greyed rows never run), and cached, so hot checks are O(1). Lists don’t Check per item: they hit the reverse index once (§04). “READ/WRITE” names the operation being authorized, not a DB write.
READ → Check(doc:contract, viewer, U)
_this · ∃ doc:contract#viewer@U
computed · editor(doc:contract)
parent↦viewer · space:pharma#viewer@U
—
WRITE → Check(doc:contract, editor, U)
_this · ∃ doc:contract#editor@U
computed · owner(doc:contract)
parent↦member · space:pharma#member@U
—

Access isn't a boolean on a grant — it's which relation you Check() for the operation: canRead resolves viewer, canWrite resolves editor. A direct doc#viewer@user grant satisfies viewer but has no path to editor — that's a read-only viewer. Space members hold member, which editor reaches via parent ↦ member, so they read and write. The hierarchy owner ⊂ editor ⊂ viewer is expressed with computed usersets, evaluated lazily with short-circuit and caching.

Cache & revoke semantics. Check results key on (object, relation, subject). A direct revoke is immediate at the source of truth — deleting the tuple removes the edge, so any uncached check is correct at once.

Invalidation: a grant/revoke purges the cache keys it directly touches. Derived access (one Space-membership change affecting many docs) is the read-side echo of the old write fan-out, so we don't fan out invalidations — we stamp a per-Space version into the key and lean on a short TTL, which bounds worst-case staleness for inherited access. Read-your-writes is guaranteed within a request by a snapshot token (Zookie). Net revoke latency: immediate for the direct edge, ≤ TTL for inherited access — a deliberate, tunable bound rather than a hidden one.

04 / Grants & queries

Writes are one row; reads use the reverse index

A single-file share, a containment edge, and a Space membership are all the same single INSERT. "What can this user see?" is one reverse-index sweep.

Ad-hoc grant and Space-derived access live in one relation. A single-file share — doc:contract#viewer@user:sam — is the same row shape as a membership, resolved by the same Check() and indexed by the same reverse index. No separate single-file-sharing subsystem, no one-object containers.

05 / Traversal depth

A flat Space boundary bounds Check() recursion

The obvious objection — "our folders nest deeply, won't auth traversal be deep too?" — doesn't apply: folders are a UI hierarchy, not edges in the auth graph. Only one doc#parent@space edge matters, so depth is constant no matter how deep the UI nests.

● ACL inheritance via parent-folder chain — O(depth)

idle

● parent↦member to a flat Space — O(1)

idle
folders still nest in the UI — they're just not edges in the auth graph

If containment recursed through the folder tree (folder#parent@folder…), Check() depth would track nesting and degrade to serial per-level lookups. Modeling parent only as doc#parent@space keeps the graph flat: one tuple-to-userset hop plus one membership lookup — bounded depth regardless of how deep the folder UI nests.

07 / The framework we'd build

A typed, app-embedded Zanzibar-lite for Drizzle

Not a separate auth service — a thin layer over the one tuple table, in our DB and our transactions. The §03 rewrites become a typed schema; everything else is a small runtime.

One tuple table

Typed entity helpers (user(), doc(), space()), schema-validated grants — no arbitrary strings, no garbage rows.

Typed schema DSL

self · direct · union · computed · tupleToUserset — the exact §03 rewrites, now compile-checked. Invalid relations don't typecheck.

Runtime API

grant · revoke · check · assert · listObjects · deleteObject, plus can(read/write/share) permission aliases.

App-embedded

Same DB, same transaction (withTx), cache hooks, tenant isolation, and an explain() trace for support tooling.

One definition → the typed client, generated for you

This is the Permix move worth copying: you write the model once, and the entity helpers, the per-entity relation names, the grant/check/listObjects signatures, and the permission aliases are all inferred from it — autocomplete and compile errors, nothing restated.

The API surface — Permix-style, entity-agnostic

The façade app code actually calls should be a typed check(entity, action) — the Permix shape. But Permix is a stateless rules layer (it stores nothing and has no listObjects); it's the front door, not the engine. The win is to keep that ergonomic, entity-agnostic API and back its rules with the ReBAC resolver — Permix's rules are predicates, which is exactly the seam.

The one thing to verify in a spike: Permix is built around synchronous, setup-time rules; the ReBAC resolver is async, per-object (it traverses the graph). Confirm Permix's async/dynamic-rule support fits — or wrap the same check(entity, action) shape in-house over authz. Either way the rule: Permix-style is the façade; tuples stay the source of truth.

Don't let "use Permix" become "compute permissions from roles" — that loses one-write shares, listObjects, cross-org, and the no-denormalization win this whole design is for.

The shape: one normalized tuple table, a typed schema, a thin runtime, and a Permix-style typed façade — bounded traversal, reverse-index listing, transaction support, cache hooks, and explain(). Not a massive auth service: a typed, app-embedded Zanzibar-lite that makes the tuple model feel native to Drizzle.

08 / Anara's two hard cases

Workspace-wide & cross-org, with primitives

Anara's permissions do two things a naive tuple table doesn't obviously cover: "everyone in the workspace can see this" (the 'workspace' sentinel) and sharing across orgs. Both fall out of standard ReBAC — using subjectRelation and tenantId, columns the schema already has.

Case A · "everyone in the workspace" → a userset subject

Today this is one chatbotPermission row with userId='workspace'. The fan-out-free equivalent grants the org's whole member set — a userset subject (subject is a relation, not a user). One row, expanded at check time.

// one row = the entire workspace, no per-member fan-out
doc:report#viewer@org:acme#member
Check(doc:report, viewer, user:dana)
  _this · doc:report#viewer@user:dana ........... no
  _this · doc:report#viewer@org:acme#member ..... userset, recurse
          Check(org:acme, member, user:dana)
            org:acme#member@user:dana ........... YES
  → ALLOW

Join or leave the org and workspace-wide access follows automatically — no doc rows touched. The same mechanism handles group grants (@group:eng#member).

Case B · sharing across orgs → a resource-tenant grant

Tuples are tenant-scoped, but a user identity isn't owned by a tenant. A cross-org share is a direct grant that lives in the resource's tenant, to a global user id.

// lives in the resource's tenant; bob's home org is beta — irrelevant
[tenant=acme] doc:contract#viewer@user:bob
Check[tenant=acme](doc:contract, viewer, user:bob)
  _this · doc:contract#viewer@user:bob .......... YES   → ALLOW

Check[tenant=acme](doc:report, viewer, user:bob)   // a workspace-wide doc
  _this · doc:report#viewer@user:bob ............ no
  _this · doc:report#viewer@org:acme#member ..... userset, recurse
          Check(org:acme, member, user:bob)
            org:acme#member@user:bob ............ NO   → DENY (bounded)

External users get only their explicit grants — no org:acme#member, no Space membership — so workspace-wide and Space-derived access never leak to them. That's the "external guest" boundary, by construction. Compliance (HIPAA, "may this org share externally?") is a policy check at grant time, not in the resolver: the tuple model permits the edge; policy decides whether it may be written.

Both reduce to columns the schema already has. subjectRelation turns "everyone in the workspace" into one userset tuple; tenantId + global user ids make cross-org a direct grant in the resource's tenant. No sentinels, no fan-out, no special cases in the resolver — just two more tuple shapes.

09 / Prior art

This is Zanzibar

Relationship-based access control is the de-facto authorization model at scale.

Google's Zanzibar backs Drive, YouTube, Photos and Cloud IAM. The open implementations are OpenFGA (CNCF), SpiceDB (Authzed) and Ory Keto. Shared machinery: a relation-tuple store, namespace configs with userset rewrites (_this, computed usersets, tuple-to-userset), a reverse index for ListObjects, a Leopard-style index to flatten deeply nested group expansion, and snapshot tokens (Zookies) for external consistency. The schema above is a single-table, MySQL-native expression of the same tuple model.

Zanzibar (OSDI '19)Google DriveCloud IAMOpenFGASpiceDBOry Keto

Invariant across all of them: persist relationships, derive permissions at read time with caching — never materialize per-object ACLs. Materialization is precisely the write-amplification and drift problem ReBAC exists to remove.
