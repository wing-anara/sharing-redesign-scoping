Authorization model

Sharing as a
relation-tuple graph

Replacing per-object ACL denormalization with relationship-based access control (ReBAC): typed relation tuples in one store, access decided by graph traversal at read time.

Thesis: stop materializing a permission row per (object, subject). Persist edges as tuples — object#relation@subject — and resolve Check(object, relation, subject) as a userset rewrite over the graph, cached.

01 — Write amplification

Denormalized ACLs fan out on every write

Today a grant materializes a permission row for every descendant object. Cost is O(subtree) per mutation.

● Denormalized ACL — row per (object, subject)

subtree of objects under the folder
folder:pharma → 24 descendant objects
permission rows written: 0

● Relation tuple — row per grant

one edge inserted into the tuple store
space:pharma — membership relation
tuples written: 0
run twice — add user:anna, then user:bob

Why it hurts: write amplification is O(subtree) per grant; re-parenting or a visibility flip triggers an unbounded subtree recompute (delete + reinsert every descendant row), and a background reconciliation job has to exist to repair drift. The tuple model writes one row per grant and derives the rest at read time.

02 — Data model

Relation tuples form an authorization graph

Three relations, one store. Membership, containment, and direct grants are the same row shape: object#relation@subject. The subject can be a user or a userset.

member parent (containment) viewer (direct grant)

Every row is a typed edge. space:pharma#member@user:wing is a membership edge; doc:report#parent@space:pharma is a containment edge whose subject is an object (used by tuple-to-userset); doc:contract#viewer@user:sam is a direct grant. Permissions aren't stored — they're paths through this graph, computed on demand.

03 — Evaluation

Check() is a userset rewrite, evaluated lazily

Pick a subject. Watch the union expand and short-circuit on the first satisfying branch.

// rewrite rule
viewer(doc) ≜ _this ∪ (parent ↦ member)
Check(doc:contract, viewer, U)expand →
∪ union_this ∪ parent↦member
[_this] ∃ doc:contract#viewer@U ?
[ttu] follow doc:contract#parent@space:pharma↦
Check(space:pharma, member, U)recurse

select a subject →

_this resolves direct grants; parent ↦ member is a tuple-to-userset rewrite — follow the parent edge to the Space, then evaluate member there. The union short-circuits: user:sam resolves at _this; user:dana only via the recursive branch; user:eve exhausts both and is denied. Hot subgraphs are cached; consistency is pinned with a snapshot token (Zookie).

04 — Direct grants

A single-file share is one leaf tuple

No new object, no subtree write — the same relation, resolved by the _this branch.

Ad-hoc grant and Space-derived access live in one relation:  doc:contract#viewer@user:sam

It's evaluated by the same Check() shown above and indexed by the same reverse index, so ListObjects(user:sam, viewer) ("what can sam see?") returns direct grants and Space-derived objects in one pass — no separate single-file-sharing subsystem, no one-object containers. Write cost is one row; read cost is one leaf lookup in the _this set.

05 — Traversal depth

A flat Space boundary bounds Check() recursion

Organizational folder nesting stays out of the authorization graph, so depth is constant.

● ACL inheritance via parent-folder chain — O(depth)

idle

● parent↦member to a flat Space — O(1)

idle
folders still nest in the UI — they're just not edges in the auth graph

If containment recursed through the folder tree (folder#parent@folder…), Check() depth would track nesting and degrade to serial per-level lookups. Modeling parent only as doc#parent@space keeps the graph flat: one tuple-to-userset hop to the Space plus one membership lookup — bounded depth regardless of how deep the folder UI nests.

06 — Prior art

This is Zanzibar

Relationship-based access control is the de-facto authorization model at scale.

Google's Zanzibar backs Drive, YouTube, Photos and Cloud IAM. The open implementations are OpenFGA (CNCF), SpiceDB (Authzed) and Ory Keto. The shared machinery: a relation-tuple store, namespace configs with userset rewrites (_this, computed usersets, tuple-to-userset), a reverse index for ListObjects, a Leopard-style index to flatten deeply nested group expansion, and snapshot tokens (Zookies) for external consistency.

Invariant across all of them: persist relationships, derive permissions at read time with caching — never materialize per-object ACLs. Materialization is precisely the write-amplification and drift problem ReBAC exists to remove.
