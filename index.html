Sharing is a pile of
sticky notes. Let's
make it rooms.

A plain-English look at why sharing is slow and fragile today, what we want instead, the tricky bits we have to decide, and the steps to get there safely.

One sentence:  today we label every single file with "who's allowed." We want to put files in rooms and just keep a guest list for each room.

01 — How it works today

We put a sticky note on everything

And that's exactly why it's slow and breaks.

Picture your files as boxes inside boxes inside boxes — a folder holds folders, which hold files. When you share the top box with a coworker, the app doesn't just remember "this box is shared." It opens every box and every file inside, all the way down, and sticks a note on each one that says "Sam is allowed here."

📌
Share a folder with 10,000 things inside → the app writes 10,000 sticky notes, one per item.
➕
Add a second coworker → another 10,000 notes.
🔁
Change your mind and make it private → peel every note back off and redo them all.
🤖
There's a cleanup robot whose only job is to fix notes that fell out of sync. The fact it needs to exist is the tell that this is fragile.

And it's not one sticky-note machine — it's three. Three different features all scribble onto the same items in slightly different ways:

1) "invite this person to this folder"  ·  2) the sticky notes themselves (what the app actually reads to decide access)  ·  3) "make this folder visible to everyone at the company." Three overlapping systems = confusing, easy to get wrong, and double the code to keep in sync.

02 — What we want instead

Rooms with a guest list

One simple idea replaces all three sticky-note machines.

📌🗄️
Today

A note on every object

To let a friend into your house you copy a permission slip onto every book, spoon, and sock. Change the guest list → re-label everything.

🚪🗝️
The plan

A key to the room

Put your stuff in a room. Keep a guest list on the door. To share, add a name to the list — the stuff inside never gets touched.

We call a room a Space. Here's the whole model:

🚪
A Space is a room. Your files live inside it.
📝
Each room has a guest list. You're either the Owner (you run the room — rename it, manage the list) or a Member (you can use everything inside).
🔑
To share, add a name to the list. That's one entry. None of the files inside change at all.
🙂
"Private" is just a room with one person in it. "Shared" is a room with more. Same thing, no special mode.
⚡
Checking access is one quick look: "what room is this file in, and is this person on that room's list?" (plus: "or was this one file handed to them directly?" — see snag 07).
📦
A file lives in exactly one room. Putting it in another room is a real move — and we'll warn you it becomes visible to that room's people.
10,000 → 1

Adding someone goes from thousands of writes to a single guest-list entry.

3 → 2

Three tangled systems become two: rooms for group sharing, a thin direct-share for one-offs.

3 → 2

Roles simplify to just Owner and Member.

03 — The snags

The tricky bits we have to decide

Switching to rooms is clean — except for these. Each one needs a call before we build.

snag 01

A folder shared two different ways at once

The problemToday you can share a big folder with Anna, and separately share one folder inside it with Bob. But a thing can only be in one room. So where does Bob go?
What we'd doPut Bob on the big room's guest list too. He can now see a bit more than before, so we'd flag every case like this in a preview report and let a human approve it.
snag 02

"Look but don't touch" disappears

The problemToday some people are view-only. The new model only has Owner and Member — and Members can edit. So view-only people would gain editing power.
What we'd doDecide with the team: either accept that view-only becomes can-edit, or keep a temporary read-only flag for the workspaces that truly rely on it. Needs a product call.
snag 03

"Everyone at the company can see this"

The problemToday a folder can be visible to the whole company automatically. Rooms are opt-in by guest list, so what happens to "everyone"?
What we'd doMake one company-wide room that everyone is automatically on, OR drop auto-visibility and make people pick rooms on purpose. Leaning toward a company-wide room.
snag 04

Moving stuff between rooms

The problemIf we stamp the room name onto every file (so reads are fast), then moving a folder to a new room means re-stamping everything inside it — the old subtree problem, but smaller.
What we'd doAccept it. Moves are rare; sharing happens constantly. Fast reads + the occasional move re-stamp is a much better trade than today's constant re-stamping.
snag 05

Not breaking anything during the switch

The problemWe can't flip a switch overnight on live data and hope. If the new system is wrong, people lose access to their stuff.
What we'd doRun both systems side by side for a while. The new one answers, the old one is a safety net. Only delete the old one once the new one has proven it gives the same answers.
snag 06

Keeping the strict-compliance rules

The problemSome workspaces have strict rules about who can be shared with and how. Those rules are load-bearing — dropping them would be a real problem.
What we'd doKeep every one of them, just re-applied at the room level instead of the folder level. Same limits, same checks, new home.
snag 07

Flinging one file at one person

The problemRooms are great for sharing a bunch of stuff with a group. But people also just want to fire off one file to one person. If every one-off makes a new room, you get a pile of silly one-file rooms — and worse, the file leaves your own space to live in that new room. Rooms are the wrong tool for this, and it'd quietly recreate the "lots of tiny relationships" problem we set out to kill.
What we'd doDon't force one-offs through rooms. Keep a thin "direct share" on the side — hand this one file to this one person, no room created, file stays where it is. Access then asks two cheap questions: "in the room?" or "handed this file directly?" So we land at two systems, not one — rooms for durable group sharing, direct-share for quick one-offs. Product call: how hard do we push people to group things into rooms instead?

04 — What we need to do

The plan to make it work

Build the new thing quietly next to the old thing, copy everyone over, prove it matches, then turn the old thing off.

  1. Create the "rooms" and "guest lists." Add the new room, membership, and invite records. Nothing changes for users yet.build quietly
  2. Teach the app to check the guest list. Add the new "is this person in the room?" check, but keep reading the old sticky notes as a backup. build quietly
  3. Give everyone a starter room. Each person gets a default personal room the moment their account is created.build quietly
  4. Build the room controls. Create / rename / archive a room, invite people, change roles — carrying over the compliance and limit rules.build quietly
  5. Make invite links work for rooms. Share-by-link, including the "join the company, then join the room" path for outsiders.build quietly
  6. Copy everyone into rooms (the big migration). A careful, repeatable job that turns today's private / team / shared folders into rooms with the right guest lists. Runs as a preview first so we can eyeball anything that changes.the switch
  7. Make "assign to a room" a proper move. With a clear "this becomes visible to N people" confirmation.the switch
  8. Reshape the sidebar and library around rooms. Group everything by room; one flat library with a room filter.the switch
  9. Unify the access logic. Replace the duplicated front-end and back-end rules with one shared "are you in the room?" source of truth.the switch
  10. Throw away the sticky notes. Once the new system has proven itself, delete the old permission mirror, its cleanup robot, and the old share screens.clean up
  11. Sunset the old public share link. Keep old links working read-only, watch how often they're used, then set a retirement date.clean up
★
Why this order: steps 1–5 ship invisibly with zero user impact. The migration (6) is the only risky moment, so it runs as a preview first and keeps the old system as a backup. Nothing old gets deleted (10–11) until the new system has been proven correct in production.
